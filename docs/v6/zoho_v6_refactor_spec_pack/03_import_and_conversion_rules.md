# 03 — Import and Conversion Rules

## Import shape

The import CSV is one row per Contact.

A single Contact row can create multiple Product Deals and multiple Quotes.

## High-level flow

```text
1. Read Lead/import row.
2. Create/reuse Account.
3. Create/reuse Contact.
4. Resolve Product Interest.
5. Parse Acquisition / Expansion / Renewal quote groups.
6. For each Product:
   create/reuse Product Deal.
   ensure Quote exists.
7. Apply quote lifecycle in order:
   Acquisition -> Expansion -> Renewal.
8. Roll up Deal Amount.
9. Roll up Account state/status.
```

## Product Deal creation

A Product Deal can be created from either:

```text
Product Interest
```

or:

```text
Quote evidence with Quote Plan Products
```

If neither exists:

```text
No Product Deal can be created.
Create Manual Review.
```

## No Product Interest gating

Commercial progression requires Product context.

Rules:

```text
MQL can exist without Product Interest.
SQL requires Product Interest or product-specific intent.
FTP requires a Product Deal and Quote.
```

Since no Deal may exist without a Quote, FTP should always have a Quote.

## No Deal without Quote

When creating a Product Deal:

```text
If actual quote values exist -> create populated Quote.
If no actual quote values exist -> create Draft Acquisition Quote.
```

If Quote creation fails:

```text
Create Manual Review.
Do not silently leave an empty Product Deal.
```

## Quote group parsing

Lead quote groups map as follows:

```text
Acquisition Quote * -> Quote Type = Acquisition
Expansion Quote * -> Quote Type = Expansion
Renewal Quote * -> Quote Type = Renewal
```

If only Product Interest exists:

```text
Create Draft Acquisition Quote.
```

If Acquisition fields exist:

```text
Create/update Acquisition Quote.
```

If Expansion fields exist:

```text
Create/update Expansion Quote.
```

If Renewal fields exist:

```text
Create/update Renewal Quote.
```

If a Renewal Quote is already implied/generated by a Closed Won Quote, reuse/update the existing Renewal Quote instead of creating a duplicate.

## Product grouping

Group quote fields by `Quote Plan Products`.

For each Product:

```text
Account × Product Deal
```

Examples:

```text
Product Interest = UX + 360
-> create UX Deal and UX Draft Acquisition Quote
-> create 360 Deal and 360 Draft Acquisition Quote
```

## Stage handling on import

Quote Stage from import controls the created Quote's stage.

Important:

```text
Confirmed = terms agreed but not signed.
Closed Won = signed.
```

Only Closed Won triggers signed lifecycle behavior.

## Import lifecycle order

When multiple quote groups exist for the same Product:

```text
1. Acquisition
2. Expansion
3. Renewal
```

Examples:

### Acquisition only

```text
Create Acquisition Quote.
If Closed Won -> create Renewal Quote.
```

### Acquisition + Expansion

```text
Create Acquisition Quote.
Apply Acquisition lifecycle if Closed Won.
Create Expansion Quote.
If Expansion Closed Won:
    Expansion supersedes active terms.
    Existing Renewal Quote is updated to match Expansion terms.
```

### Current-only / Renewal-only rows

The final CSV structure will define exact handling. If a row contains Renewal fields without a prior Acquisition or active quote, create Manual Review unless the data can be safely treated as Acquisition.

## One row per Contact with multiple Products

If one Contact has multiple Product Interests:

```text
Create one Product Deal per Product.
Create one Quote per Product Deal.
Set Quote.Contact to the Contact when appropriate.
```

## Contact association

Contact belongs to a Product Deal if:

```text
Contact Product Interest includes Deal Product
or Contact is Quote.Contact
or Contact was manually added to Deal Contact Roles
```

Do not attach every Account Contact to every Product Deal.

## Decision Maker activation after import

Automatic activation task creation is only for Decision Makers by default.

Non-Decision Makers may be manually activated.

## Fresh import assumption

No production migration logic is required.

Existing Zoho data will be cleared before import.

Test records must use a unique E2E identifier and be cleaned up after the test.
